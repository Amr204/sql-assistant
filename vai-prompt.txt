1. جميع الجداول.
2. جميع الأعمدة.
3. أنواع البيانات.
4. Primary Keys.
5. Foreign Keys.
6. Unique Constraints.
7. Indexes.
8. Views.
9. Stored Procedures إن وجدت.
10. العلاقات الصريحة.
11. العلاقات المحتملة inferred relationships.
12. الجداول المحورية.
13. جداول facts.
14. جداول dimensions.
15. جداول lookup/reference.
16. جداول audit/logs.
17. جداول users/auth/security.
18. جداول configuration.
19. جداول transactions.
20. أعمدة التواريخ المهمة.
21. أعمدة الحالة Status/State/Type.
22. أعمدة المبالغ Amount/Balance/Total.
23. أعمدة حساسة PII/secrets.
24. أعمدة يمكن استخدامها للفرز الافتراضي.
25. معنى كل جدول business meaning.
26. grain لكل جدول: ماذا يمثل الصف الواحد؟
27. أشهر join paths.
28. المرادفات العربية والإنجليزية المتوقعة لكل جدول وعمود.
29. أسئلة المستخدمين المحتملة.
30. مؤشرات KPIs المحتملة.
31. مخاطر SQL أو privacy داخل القاعدة.

إذا كان المعنى غير مؤكد، لا تخترع بثقة. اكتب:

reason: "inferred from table/column names"

==================================================
9. Secure SQL Execution
==================================================

لا تعتمد على الـ prompt فقط للأمان.
أضف PromptInjectionFilter أو InputSafetyClassifier بسيط:
- يرصد طلبات تجاوز السياسات
- يرصد "ignore previous instructions"
- يرصد طلب استخراج schema كامل بدون صلاحية
- يرصد طلبات بيانات حساسة
- لا يعتمد عليه وحده للأمان، بل كطبقة قبل SQL policy
ابنِ SecureRunSqlTool أو wrapper حول RunSqlTool يقوم بالآتي قبل التنفيذ:

- يسمح فقط بـ SELECT.
- يمنع INSERT/UPDATE/DELETE/MERGE/DROP/ALTER/TRUNCATE/EXEC.
- يمنع multiple statements.
- يمنع comments المشبوهة.
- يمنع sys و INFORMATION_SCHEMA إلا لو admin ومصرح.
- يمنع OPENROWSET و OPENQUERY و xp_cmdshell.
- يمنع cross-database queries.
- يمنع SELECT *.
- يفرض table allowlist.
- يفرض column allowlist أو يمنع sensitive columns.
- يطبق max row limit.
- يضيف TOP افتراضي عند الحاجة.
- يضيف query timeout.
- يسجل audit log لكل محاولة.
- يرجع error واضح وآمن للمستخدم بدون تسريب معلومات حساسة.

استخدم parser مناسب مثل sqlglot إن أمكن مع T-SQL support، ولا تعتمد فقط على regex.

25. User Resolver
==================================================

نفذ UserResolver بثلاثة أوضاع:

1. dev:
   مستخدم افتراضي من env.

2. header:
   يقرأ:
   - X-User-Id
   - X-User-Email
   - X-User-Groups

3. future_oidc:
   interface جاهز لاحقًا لـ JWT/OIDC.

في production:
- لا تقبل admin group من headers غير موثوقة.
- وثّق ذلك في SECURITY.md.
